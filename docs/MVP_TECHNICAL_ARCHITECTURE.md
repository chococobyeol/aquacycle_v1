# 초기 MVP 기술 구조

> **기술 구조 기록:** 현재 다섯 개 미션의 구체적인 구현 기준은 [AquaCycle V2 재설계 명세](./V2_REDESIGN.md)를 우선한다.

> 문서 상태: 기술 방향 확정·현재 구현 반영  
> 작성일: 2026-07-13 · 최근 갱신: 2026-07-22
> 대상 범위: 다섯 개 미션, 체리새우·미생물 물질 순환을 포함한 실험실 모드와 향후 관찰용 위젯 확장

> 현재 공간 수온·부력 유동·보존형 공통 물질 수송 구조는 [생태·수질·열수송 모델 근거와 구현 기준](./SIMULATION_MODEL_AND_REFERENCES.md)에 기록한다. 해당 문서를 현재 구현의 우선 기준으로 사용한다.

## 1. 선택한 기술 조합

초기 MVP는 다음 조합으로 구현한다.

| 역할 | 기술 |
|---|---|
| 데스크톱 앱과 창 관리 | Electron |
| 언어 | TypeScript |
| 빌드·개발 서버 | Vite |
| 미션·실험실 UI | React와 HTML/CSS |
| 수조·조류·체리새우 2D 렌더링 | PixiJS WebGL |
| 돌 중력·충돌·회전·안착 | Matter.js |
| 생태 시뮬레이션 | TypeScript Web Worker |
| 패키징 | Electron Forge |

Godot과 Tauri는 현재 구현 기술로 사용하지 않는다.

## 2. Electron을 선택한 이유

- Windows와 macOS에서 동일한 Chromium 계열 렌더링 환경을 함께 배포할 수 있다.
- 일반 게임 창과 향후 위젯 창을 같은 `BrowserWindow` 구조로 관리할 수 있다.
- 테두리 제거, 항상 위, 작업 표시줄 숨김과 입력 통과 같은 위젯용 창 기능을 제공한다.
- 미션 UI, 실험실 설정, 분석 그래프와 접근성 처리를 HTML/CSS로 구현할 수 있다.
- TypeScript로 시뮬레이션, 렌더링과 UI를 함께 작성하여 자료 조사·자동 테스트·디버깅 흐름을 통일할 수 있다.
- PixiJS와 Matter.js가 초기 MVP에 필요한 2D 메시 렌더링과 강체 물리를 제공한다.

Electron은 Chromium 기반 다중 프로세스를 포함하므로 Tauri보다 상시 실행 비용이 커질 가능성이 있다. 이를 이유로 시뮬레이션과 렌더링의 유휴 비용을 초기부터 측정하고, 위젯 모드에서는 표현 세부 수준과 갱신 빈도를 낮춘다.

## 3. Tauri를 선택하지 않은 이유

Tauri 2는 테두리 없는 창, 항상 위, 투명 창, 입력 무시와 Windows·macOS 패키징을 지원하므로 기능 부족으로 제외한 것은 아니다.

다만 Tauri는 운영체제의 WebView를 사용하므로 Windows에서는 WebView2, macOS에서는 WKWebView에서 실행된다. 이 게임은 PixiJS WebGL, 사용자 정의 셰이더, 투명·소형 창과 입력 처리를 함께 사용하므로 플랫폼별 렌더링 차이를 별도로 검증해야 한다.

또한 macOS의 일부 투명 창·Dock 동작은 Rust 또는 플랫폼별 API 처리가 필요하다. Rust 도구 사용 자체는 가능하지만, 초기 MVP에서 얻는 실행 크기상의 이점보다 구현·검증 경로가 갈라지는 비용이 더 크다고 판단한다.

## 4. 프로세스와 책임 분리

```text
Electron Main Process
├─ 일반 창·향후 위젯 창 생성과 상태 전환
├─ 저장 파일과 앱 생명주기
└─ 최소한의 안전한 IPC

Renderer Process
├─ React: 미션·실험실·분석 UI
└─ PixiJS: 수조·구조물·조류·균 필름·체리새우 리그·광량/수질 오버레이 렌더링

Simulation Web Worker
├─ 고정 시간 간격 시뮬레이션
├─ Matter.js 돌 물리
├─ 환경 셀과 광량장
├─ 표면 구간별 조류 생체량과 확산
├─ 체리새우의 운동학적 이동, 섭식, 에너지와 생애 주기
├─ 시나리오별 생지화학 ledger와 네 수질의 공간 확산
├─ 표면 구간별 분해균·질산화균 필름 반응과 확산
└─ 미션 판정용 상태 산출
```

Electron 메인 프로세스에서는 생태 시뮬레이션이나 물리 계산을 수행하지 않는다. 렌더러는 입력 명령을 Worker에 보내고, Worker가 반환한 상태 스냅샷을 화면에 표시한다.

도전 모드와 실험실은 동일한 시뮬레이션 코어를 사용한다. 일반 도전 미션은 시작 뒤 배치를 잠그지만, 미션 5는 진행 중 환경을 읽고 대응하는 과제이므로 일시정지 상태에서 균 필름 접종만 예외적으로 허용한다. 구조물·조류·체리새우는 계속 잠긴다. 실험실은 일시정지 중 구조물·조류 접종체·체리새우·균 필름 편집 명령을 허용한다.

## 5. 향후 위젯 확장 원칙

향후 위젯은 별도의 게임을 실행하지 않고 같은 저장 상태와 시뮬레이션 스냅샷을 표시한다.

- 테두리 없는 작은 `BrowserWindow`로 생성한다.
- 항상 위와 작업 표시줄·Dock 노출 방식을 플랫폼별로 설정한다.
- 기본적으로 관찰만 제공하며 편집은 일반 창으로 전환한다.
- 미세한 조류 스탬프와 입자 수를 줄이고 화면 갱신 빈도를 낮춘다.
- 완전 투명 데스크톱 펫 형태보다 직사각형 또는 둥근 모서리의 작은 수조 패널을 우선 검증한다.

## 6. 구현 원칙

- 생태 규칙과 미션 수치는 코드에 직접 넣지 않고 데이터로 분리한다.
- 시뮬레이션은 화면 프레임률과 분리된 고정 시간 간격으로 진행한다.
- 렌더링 좌표와 시뮬레이션 좌표를 분리한다.
- 구조물의 충돌 형태와 성장 표면은 스프라이트와 별도 데이터로 관리한다.
- 조류 군락 타입과 이동 동물 타입을 분리해 기존 생산자 계산에 동물 필드를 억지로 섞지 않는다.
- 돌만 Matter.js 강체로 계산하고, 체리새우는 연속 좌표·속도와 조향 규칙을 가진 운동학적 개체로 계산한다.
- 같은 초기 상태와 입력으로 재현 가능한 결과를 만들고, 필요한 난수는 시드로 관리한다.
- PixiJS, Matter.js와 Electron 의존 코드는 어댑터 경계에 모아 생태 코어와 분리한다.

## 7. 체리새우와 미생물 물질 순환

### 7.1 고정 주기 분리

현재 Worker는 역할에 따라 계산 주기를 나눈다.

```text
돌 물리                 약 60 Hz 고정 스텝
체리새우 이동·조향      기본 30 Hz, 32배 이상 10 Hz 시뮬레이션 스텝
수온·조류·새우 생태     기본 0.25초, 32배 이상 1초 시뮬레이션 스텝
전체 상태 스냅샷        약 0.20초 간격
개체 위치 motion 메시지 자동 이동 약 20 Hz, 직접 조작 최대 약 33 Hz
조류 래스터 갱신        8배 이하 약 4 Hz, 16배 이상 약 2 Hz
```

시간 배속은 `×1, ×2, ×4, ×8, ×16, ×32, ×64`를 제공한다. `×32` 이상은 장기 관찰용 빠른 진행이므로 생태·조향 스텝을 묶어 계산하되 같은 시뮬레이션 시간을 빠뜨리지 않는다. 고배속에서도 이동을 모두 계산한 뒤 생태를 몰아서 계산하지 않고, 두 단계의 예정 시각 순서대로 교차 실행해 실제로 먹이에 도착한 뒤에만 섭식이 일어나게 한다. 서로 다른 구조물 표면 사이의 전파 연결은 셀마다 가장 가까운 네 구간까지만 허용해, 돌을 많이 겹쳤을 때 이웃 수가 폭증하지 않게 한다. 조류 표시는 계산과 분리해 갱신 빈도만 낮추므로 새우 리그의 화면 애니메이션은 계속 렌더 프레임으로 움직인다.

생태 틱은 기존 미션과 물질 순환 미션이 같은 코어를 사용하되, 시나리오별 효과 상태를 따른다. 미션 1~4는 수질 되먹임이 꺼져 기존 조류·새우 결과를 유지한다. 미션 5와 실험실은 `수온 → 조류 성장·경쟁·확산 → 생물의 생산·소비 기록 → 새우 섭식·에너지·수질 스트레스·생애 주기 → 네 수질과 두 균 필름 반응·확산 → 균 필름의 표면 확산 → 미션 판정` 순서로 진행한다. 같은 생태 틱 안의 입력은 버퍼에 기록한 뒤 일괄 반영해 순회 순서가 결과를 바꾸지 않게 한다.

### 7.2 개체 상태와 렌더링

Worker는 체리새우마다 위치·속도, 방향, 생활 단계, 성별, 나이, 개체별 수명, 에너지, 행동, 먹이 목표, 행동 타이머, 성장·포란 상태, 개체별·세션별 누적 섭식량과 결정적 난수 시드를 보유한다. 평상시 먹이 탐색은 현재 위치에서 반경 `64` 안에 실제 조류가 `0.04` 이상 남은 표면 중 가까운 곳을 우선하며, 에너지가 `0.35` 이하인 약한 개체만 반경 `520`의 비상 탐색을 사용한다. 접종 위치를 별도 먹이점으로 취급하지 않으므로 새우는 성장·확산한 어느 표면에서든 조류를 만날 수 있다. 여러 개체가 같은 셀을 노리면 혼잡도를 반영하고, 섭식 시 해당 셀의 붓뚜껑말·규조류 생체량을 실제 섭취량만큼 감소시킨다.

먹이 탐색은 에너지 `0.48`에서 시작해 `0.70`까지 이어지는 히스테리시스를 사용한다. 한 번의 섭식 세션은 3~5초 또는 누적 섭취 생체량 `1.4` 중 먼저 도달한 때 끝나며, 충분히 먹고 세션을 끝낸 개체는 같은 군락의 기억 목록을 만드는 대신 기존 행동 타이머만 재사용해 2.5~4초 동안 먹이 탐색을 끄고 실제로 배회한다. 그 뒤 다시 먹이 탐색을 허용한다. 기본 섭식률은 시뮬레이션초당 `0.375`이며 셀의 먹이 밀도로 조절되고, 실제 섭취량의 `0.08`만 에너지로 바뀐다. 섭취량을 화면에서 분명히 확인할 수 있도록 생체량 제거율을 키운 대신 에너지 전환율을 함께 보정해 먹이가 많은 수조에서는 번식할 잉여 에너지가 생기고 희박한 수조에서는 개체군이 감소하게 한다.

어린 개체의 성장 진행도는 에너지 `0.44` 이상이며 최근 12초 안에 섭식했을 때만 증가하고, 180초의 나이·급식 성장 조건을 모두 만족해야 성체가 된다. 번식 후보는 암수 모두 에너지 `0.34` 이상, 양쪽 모두 최근 12초 안에 실제로 섭식했고 각 개체의 휴지기가 끝난 조건을 만족해야 한다. 같은 수조에 조건을 만족한 암수가 있으면 3초 교미 뒤 포란에 들어간다. 포란 75초는 에너지 `0.30` 이상이며 최근 24초 안에 섭식한 동안에만 진행된다. 출산은 2~3마리이고, 완전한 한 배를 만들 물질이 모일 때까지 출산을 기다린다. 출산 뒤 암컷 160초·수컷 45초, 새 성체 120초의 휴지기를 적용한다.

개체별 수명은 `900~1350` 시뮬레이션초 사이에서 결정적 난수로 정한다. 나이가 수명에 도달하면 에너지와 무관하게 자연사한다. 먹이가 계속 충분해 출생이 사망보다 많으면 개체군은 계속 증가하며, 먹이 소비가 생산량을 넘으면 섭식·성장·포란이 멈추고 굶주림과 자연사로 감소한다. 따라서 평형 개체 수는 숨은 수용량 식이 아니라 플레이어가 만든 조류 생산량과 새우의 소비·생애 주기에서 나타난다.

생태 규칙으로 사용하는 최대 개체 수나 번식 수용량 식은 두지 않는다. 다만 손상된 저장 상태나 극단적인 장기 실행이 한 번에 무한한 개체를 할당하지 못하게 `2,048`마리에서만 마지막 부화 안전선을 적용한다. 이는 정상 플레이의 평형을 만드는 규칙이 아니라 메모리 보호용 기술 가드이며, 그 전까지 번식 여부는 에너지, 최근 실제 섭식 여부, 암수 존재, 포란과 휴지기로만 결정된다.

조류의 이웃 확산도 새 생체량을 복제하지 않는다. 각 틱의 시작 상태에서 정착 제안을 계산하고, 도착 셀의 빈 공간과 출발 셀이 내보낼 수 있는 생체량으로 모든 이동량을 비례 제한한 뒤 출발 셀에서 뺀 양과 같은 양을 도착 셀에 동시에 더한다. 광합성에 따른 국소 성장만 새 조류 생체량을 만들며, 단순한 표면 확산은 전체 생체량을 보존한다.

초기 지급 성체는 번식 단계가 동기화된 실험 집단으로 보지 않고 첫 번식 휴지기를 개체별 `100~400`초에 결정적으로 분산한다. 이후에는 일반 개체와 같은 먹이·에너지·암수·휴지기 규칙만 사용한다. 표준 미션 4 회귀 장면은 300초에 5~8마리, 600초에 9~13마리 범위로 증가해야 하며, 이 검증 범위는 고정 개체 수 식이 아니라 실제 섭식과 서로 다른 첫 번식 시점이 소비자 폭증을 완충하는지 확인하는 기준이다.

PixiJS는 시뮬레이션의 위치와 행동 상태를 받아 부위별 체리새우 리그를 보간하고, 더듬이·꼬리·다리의 작은 움직임을 렌더 프레임에서 만든다. 애니메이션 변형은 생태 위치나 섭식 판정을 바꾸지 않는다. 새우는 Matter.js 강체가 아니며 정교한 유체역학이나 개별 다리 충돌을 계산하지 않는다.

에너지가 0이 되거나 개체 수명에 도달한 새우는 생존 목록과 미션 성체 수에서 즉시 제외되고, Worker는 사망 위치·방향·몸길이·생활 단계·사망 원인과 경과시간을 가진 사체 스냅샷을 별도로 만든다. PixiJS는 같은 절차형 리그를 저채도·정지·말린 자세로 재사용해 최대 55 시뮬레이션초 동안 표시하고 후반부에 흐리게 만든다. 이 시각 수명은 ledger의 사망 질량을 다시 더하거나 제거하지 않는다.

스냅샷은 개체별 누적 섭식량과 수조 전체 누적 섭식량도 전달한다. 관찰 UI는 이를 개체 정보에 표시하고, 일정 간격으로 전체 조류량과 성체·어린 체리새우 수를 샘플링해 짧은 시간 추세 그래프로 보여준다. 따라서 섭식 연출만으로 추측하지 않아도 실제 조류 감소와 개체군 변화를 확인할 수 있다.

### 7.3 시나리오별 생지화학 ledger

생지화학 ledger는 `36 × 20` 고정 크기 공간 구획과 공간 물속 무기탄소·닫힌 공기층 저장고를 사용한다. 모든 미션에서 같은 인터페이스를 쓰되, 실제 수질장과 보존 장부를 갱신하는지는 시나리오 데이터로 결정한다.

- 조류 생체량과 광량에 따른 잠재 산소 생산량
- 조류와 체리새우의 잠재 산소 요구량
- 새우가 실제로 먹은 양에 따른 용존 배설물 발생량
- 새우가 죽은 위치의 사망 질량과 고형 유기물량
- 유기물, 독성 노폐물, 영양염과 용존산소의 위치별 농도 및 확산
- 유한한 물속 무기탄소, 공기층 이산화탄소와 공기층 산소
- 생물·유기물·사체·균 필름을 포함한 전체 질소·탄소 보존 오차

미션 1~4에서는 `effectsEnabled = false`다. 이 범위에서는 ledger가 관찰용 입력만 누적하고 네 수질의 확산, 균 반응, 산소 부족·독성 피해와 영양염 제한은 생태에 되먹임되지 않는다. 미션 5와 실험실에서는 `effectsEnabled = true`이며 다음 연결이 실제 상태를 바꾼다.

```text
조류 + 섭식                 → 새우 몸 + 고형 배설물 + 호흡 산물
사체·죽은 조류·균 필름      → 고형 찌꺼기 → 유기물
유기물 + 산소 + 분해균       → 독성 노폐물 + 분해균 성장
독성 노폐물 + 산소 + 질산화균 → 영양염 + 질산화균 성장
영양염 + 무기탄소 + 빛       → 조류 성장 + 산소 생산
낮은 산소·높은 독성 노폐물   → 체리새우 건강 저하와 사망
```

두 균은 개별 세균이 아니라 각 성장 표면 구간의 `0~1` 생체량으로 저장한다. 섭취 속도는 해당 먹이와 산소의 반포화식으로 제한하고, 실제로 소비할 수 있는 농도보다 많은 양을 처리하지 않는다. 처리량의 일부만 새 필름으로 전환하며, 먹이·산소 활동도가 떨어질수록 기아 사멸이 커진다. 분해균은 유기물 펄스에 빠르게 증가한 뒤 고갈되면 감소하고, 질산화균은 더 느리지만 낮은 부하에서 오래 유지된다. 두 필름은 같은 표면 수용력을 공유하고, 과밀하면 탈락하며, 인접 표면 확산과 질량 보존형 수중 전파·재정착으로 떨어진 표면까지 퍼진다.

수면에서는 용존산소와 물속 무기탄소가 각각 유한한 공기층 산소·이산화탄소 저장고와 완만하게 교환된다. 외부 기준 농도에서 무한 보충하지 않는다. 유기물·독성 노폐물·영양염·산소·무기탄소는 성분별 국소 확산과 공간 수온에서 계산한 하나의 보존형 공통 경계 유량을 함께 사용한다. 모든 농도와 필름량은 유한한 비음수 범위로 제한하며, 매 스냅샷에서 기준 시점 대비 질소·탄소 오차를 계산한다.

### 7.4 미션과 실험실 데이터

미션 데이터는 조류 접종 예산, 이동 동물 예산과 균 필름 접종 예산을 별도로 관리한다. `allowedSpecies`는 조류, `allowedAnimals`는 체리새우, `waterCycle.allowedMicrobes`는 균 기능군을 의미하며, 허용 여부와 시작 필수 배치를 구분한다. 네 번째와 다섯 번째 미션 모두 두 조류와 체리새우를 허용하지만 필수 접종·필수 구조물·필수 새우 설치 조건은 두지 않는다.

다섯 번째 미션은 살아 있는 체리새우가 한 마리 이상인 군집을 `1,500` 시뮬레이션초 동안 연속 유지하면 성공한다. 이는 공급 성체의 최대 압축 수명보다 길어 세대교체를 요구한다. 수질의 특정 안전 구간, 접종 횟수·위치, 구조물 수와 균 필름량은 직접 채점하지 않는다. 군집이 0이 되면 연속 생존 시간은 0으로 돌아가며, 제한시간 `1,800`초 안에 목표를 달성해야 한다. 초기 영양염은 `12`로 제한해 분해·질산화 없이 시작 저장고만 소비하는 표준 대조군이 목표 전에 소멸하게 한다. 성공 또는 실패를 기록한 뒤에도 Worker의 재생 상태는 바뀌지 않고 생태계는 계속 진행한다.

실험실에서는 구조물, 두 조류, 체리새우와 두 균 필름을 무제한으로 제공한다. 미션 5와 같은 네 수질·두 균 필름·산소 및 독성 스트레스 규칙을 사용하고, 실행 후에는 일시정지했을 때만 배치를 편집할 수 있다.

### 7.5 관찰 오버레이와 수질 탐침

수질 효과가 활성화된 미션 5와 실험실은 여섯 관찰 채널을 제공한다.

- 유기물
- 독성 노폐물
- 영양염
- 용존산소
- 분해균 필름
- 질산화균 필름

선택한 채널은 관찰 지도에서 수조 위의 공간 오버레이로 표시한다. 별도 수질 탐침의 포인터 미리보기와 설치형 수질 측정점은 같은 Worker 스냅샷을 읽으며, 네 수질의 현재 수치, 두 필름의 국소 점유율과 예상 순성장률을 보여준다. 설치한 측정점은 시뮬레이션 중 계속 갱신한다. 관찰 패널은 수조 평균 네 수질·고형 찌꺼기·두 필름 총량·물속 무기탄소·공기층 기체·질소와 탄소 장부 오차 및 세 묶음의 시간 변화 그래프를 제공한다.
